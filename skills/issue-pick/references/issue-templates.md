# Issue 模板

这些模板用于 `issue-flow` 中的 Issue 草稿准备。

原则：

- 先生成草稿，再由用户确认
- 最终创建必须使用 `gh issue create --web`
- 一个 Issue 对应一个明确可交付结果
- 大需求优先拆成父 Issue 加子 Issue，或用 Milestone 聚合
- 能用 GitHub 原生 issue type、issue fields、sub-issues、dependencies 时，优先用原生能力，不把结构化元数据硬塞进正文

## 单个可交付 Issue

```markdown
## 目标

<一句话说明这次交付要达成什么结果>

## 背景 / 范围

- 背景：<为什么要做>
- 本次包含：<本轮明确要交付什么>
- 本次不包含：<明确不做什么；没有可写"无">

## 验收标准

- [ ] <可验证条件 1>
- [ ] <可验证条件 2>

## 技术约束

<会影响实现方式的限制、依赖或排除项；没有可写"无">

## 风险 / 依赖

- 风险：<风险，或"无">
- 依赖：<依赖，或"无">

## 相关文档

<需要更新的文档，或"无">
```

## 父级跟踪 Issue

适用于较大需求，需要拆成多个子 Issue 并持续汇总进度。

```markdown
## 目标

<一句话说明整个大需求的最终结果>

## 范围

- 包含：
  - <范围 1>
  - <范围 2>
- 不包含：
  - <非目标 1>

## 子任务

- 优先使用 GitHub 原生 sub-issues / dependencies 维护父子和前置关系
- [ ] #<子 Issue 编号> <子任务标题>
- [ ] #<子 Issue 编号> <子任务标题>

## 验收标准

- [ ] <整体层面的可验证条件 1>
- [ ] <整体层面的可验证条件 2>

## 风险与依赖

- 风险：<风险，或"无">
- 依赖：<依赖，或"无">
```

## 子 Issue

适用于父 Issue 已存在时的可独立执行子任务。

```markdown
## 目标

<一句话说明该子任务交付结果>

## 父级关联

- 优先使用 GitHub 原生 parent issue / sub-issue 关系
- 如需在正文补充说明：Parent: #<父 Issue 编号>

## 验收标准

- [ ] <子任务可验证条件 1>
- [ ] <子任务可验证条件 2>

## 技术约束

<仅写影响本子任务实现的约束；没有可写"无">

## 风险 / 依赖

- 风险：<风险，或"无">
- 依赖：<依赖，或"无">

## 需要同步给父 Issue 的内容

<进度、风险、验证结果等>
```

## 标题建议

- 功能：`支持用户使用邮箱登录`
- 修复：`修复未登录状态下页面跳转死循环`
- 优化：`减少批量导入时的数据库写放大`
- 杂项：`清理过时的构建配置`

标题要表达结果，不要只写动作。分类优先交给 GitHub issue type、labels 或 issue fields，而不是强制在标题前加 `feat:`、`fix:`。

不推荐：

- `更新接口`
- `处理登录`
- `修一些 bug`

推荐：

- `支持用户使用邮箱登录`
- `修复未登录状态下页面跳转死循环`
- `清理过时的构建配置`

## 分支命名建议

- 默认：`<type>/<N>-<slug>`
- 例如：`feature/123-email-login`
- 修复：`fix/456-redirect-loop`
- 维护：`chore/789-clean-build-config`
- 如果仓库已有强约束，也可退回到仓库既有格式，例如 `issue-123-email-login`

推荐把 issue 编号保留在分支名里，便于和 GitHub 的 linked branch、PR、搜索和自动化规则对应。
